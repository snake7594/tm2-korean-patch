## 2. mainmenu.obc 구조 (8bpp)
```
[0x000] 8바이트 헤더
[0x008] 팔레트 448바이트 (224색 × BGR555 2바이트)
[0x1C8] 8bpp 타일 데이터 (= 오프셋 456 부터)
```
- 타일 = 8×8 픽셀, 1픽셀 = 1바이트(팔레트 인덱스), 타일 1개 = 64바이트.
- 글자색 인덱스 = **37** (게임에서 흰색).

### 워드(단어) 배치
- 한 "워드" = 두 글자 = 32×16 = **8타일** (가로 4타일 × 세로 2타일).
- 32폭(4타일) 텍스처에서 워드는 아래로 연속 배치됨.
- 워드 W의 타일 = `8W .. 8W+7`.
  - 윗줄(4타일): `8W, 8W+1, 8W+2, 8W+3`
  - 아랫줄(4타일): `8W+4, 8W+5, 8W+6, 8W+7`
- 픽셀 (x,y) → 타일 `8W + (y//8)*4 + (x//8)`, 타일 내부 `(x%8, y%8)`.
  - 왼쪽 글자 = x 0~15, 오른쪽 글자 = x 16~31.

## 3. command_icon 구조 (4bpp)
- 헤더/팔레트 없음. **순수 타일 데이터** 256바이트 = 8타일.
- 타일 = 8×8, 4bpp(1바이트=2픽셀, 하위 니블=왼쪽 픽셀), 타일 1개 = 32바이트.
- 32×16(두 글자) = 8타일, 배치는 mainmenu 워드와 동일(가로 4타일 × 세로 2타일).

## 4. LZ10 — VRAM 안전 주의
- 게임 메뉴 그래픽은 BIOS LZ77UnCompReadNormalWrite16bit 로 **VRAM**에 풀립니다.
- 이때 `displacement<2` 백레퍼런스는 16비트 쓰기 버퍼와 충돌해 타일/팔레트가
  깨집니다(초록색 깨짐). **원본 압축을 분석하면 최소 disp=2** 입니다(disp=1 없음).
- 이 키트는 안전하게 **매치를 전혀 쓰지 않는 무손실 인코딩**(`tools.lz10_store`)을
  씁니다. 파일은 약 +12.5% 커지지만 어떤 디코더에서도 정확히 풀립니다.
- 더 작게 만들고 싶으면 `disp>=2`만 허용하는 그리디 압축기를 직접 작성해
  대체할 수 있습니다(원본과 동일한 제약).

## 5. 롬 기록 — 추가 후 FAT 재지정 (헤더 무손상)
1. 패치한 파일을 재압축.
2. 모든 FAT 엔트리의 최대 end 를 찾아 0x200 정렬 후, 그 자리(롬의 0xFF 패딩)에
   압축 데이터를 기록. (패딩이 모자라면 파일 끝을 확장)
3. `FAT[fid]`의 start/end(8바이트)만 추가 위치로 변경.
4. 헤더(0x00~0x15D)는 건드리지 않으므로 헤더 CRC가 유지됨.
   - FAT는 헤더 밖에 있고, 총용량 필드는 정보성이라 손대지 않아도 됨.

## 7. 시작 선택 메뉴(起転) — CMP 포맷

부팅 시 '새 게임(起)/이어하기(転)' 선택 화면. `kiten_patch.py` 담당.

### 파일 위치 (FNT 이름으로 탐색)
| 파일 | 경로 | 내용 |
|---|---|---|
| 타일셋 | `/uno_all/kiten/bg00/char00.chr` | CMP, 4bpp, 247타일 |
| 타일맵 | `/uno_all/kiten/bg00/scrn00.scr` | CMP, 32×32 u16 |
| 팔레트 | `/uno_all/kiten/bg00/palt.pal` | CMP, 256색(뱅크0만 사용) |

> scrn01.scr은 비어 있음(미사용). 캐릭터 스프라이트는 `kiten/sd/`(manj=卍丸 등).

### CMP 압축 포맷
```
[0] "CMP"          3바이트 시그니처
[3] ver            1바이트 = 압축 해제 '단계 수' (게임이 이 횟수만큼 BIOS 해제)
[4] f1  u32        최종 해제 크기
[8] f2  u32        외부 DS 압축 헤더 (하위바이트=타입, 상위3바이트=그 단계 출력크기)
[12] 압축 데이터 …
```
- **외부 타입**(f2 하위바이트): `0x24`=Huffman 4bit, `0x28`=Huffman 8bit,
  `0x30`=RLE, `0x10`=LZ10, `0x11`=LZ11.
- **다단계 체인**: 외부 단계를 풀면 그 결과가 또 표준 DS 압축(헤더 타입)일 수 있어,
  **f1 크기에 도달할 때까지** 각 레이어를 그 헤더 타입대로 반복 해제.
- 예) char00: ver2 = Huffman4 → LZ10 (2단계). scrn00: ver3 = Huffman4 → LZ10 → RLE.
- `ver` ↔ 단계수가 정확히 일치(전 파일 확인). **단일 RLE로 재압축하면 ver=1**.

### 디코드 후 구조
- `.chr`: 4바이트 서브헤더(= 타일데이터 크기 u32) + 4bpp 타일(32B/타일).
- `.scr`: 4바이트 서브헤더(`01 00 00 08`) + u16 타일맵(타일0-9, h플립10, v플립11, 뱅크12-15).
- `.pal`: 4바이트 서브헤더 + BGR555 색.

### 팔레트(뱅크0) 주요 인덱스
| idx | 용도 | idx | 용도 |
|---|---|---|---|
| 0 | 투명(초록) | 7 | **초록 강조**(말풍선 起/転) |
| 1 | 카드 글자(검정) | 12 | 말풍선 배경(연파랑) |
| 4 | 말풍선 본문(어두운 보라) | 14 | 起 카드 배경(분홍) |
| 6 | 본문 음영 | 15 | 転 카드 배경(어두운 보라) |
| 9·11·13 | 카드 금테 | 2·3·5 | 카드 적색 테두리 |

### 편집 방식 (`kiten_patch.py`)
1. char00+scrn00+palt 해제 → 화면(256×256) 인덱스 이미지로 렌더(뱅크0).
2. 말풍선 배경(12)의 bbox 안 텍스트색(1·2·3·4·5·6·7·8·10)을 12로 지움 →
   갈무리14로 한국어 그림(본문=4, 「처음」「계속」=초록7).
3. 起/転 카드: 각 배경색(14/15) bbox를 배경색으로 채워 한자 제거 → 처음/계속(검정1) 가운데 배치.
4. 8×8 타일로 재구성(플립 디덥). **원본 타일 수(247)까지 0-타일 패딩**해 char00
   크기를 원본과 동일하게 유지(게임이 고정 크기 복사해도 안전).
5. char00/scrn00을 **단일 RLE(ver=1)** CMP로 재압축 → 0xFF 패딩에 추가, FAT만 변경.

### 재압축이 RLE(ver=1)인 이유
- RLE 외부 압축은 게임의 char01/palt(둘 다 ver1 RLE)가 쓰는 검증된 형식이라
  같은 폴더의 동일 로더가 그대로 처리.
- Huffman 인코더를 구현하지 않아도 되고, RLE는 무손실·단순.
- 처음에 원본 ver(2/3)를 두면 게임이 2~3단계 해제를 시도하다 멈추므로 **ver=1 필수**.
